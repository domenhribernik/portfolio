1. How search works in 2026

Google still runs the same four-stage pipeline: crawl (fetch the HTML), render (run the JS in a headless Chromium, on a queue that can lag hours or days behind the crawl), index (store what survived), and rank (order it against a query). Two things follow from the render queue. First, anything that only exists after JS runs is indexed late and unreliably. Second, most crawlers that are not Google (Bing partially, and nearly every AI scraper) do not render JS at all: they see the raw HTML and nothing else.

That is why this site prerenders its JS-built content. The homepage projects section and the blog post list carry a static fallback in the HTML itself, and every blog post exists as a real static page. The interactive views (nebo, flowers, workout) stay apps, but their heads carry complete metadata so even a non-rendering crawler knows exactly what each page is.

The second index that matters now is the AI answer layer: ChatGPT, Perplexity, and Google's AI Overviews assemble answers from pages they can fetch and quote. They favor plain, well-structured HTML with clear statements of what a thing is. Section 8 covers this; the short version is that the same work that makes classic SEO good makes AI citation possible.

2. Technical SEO

One canonical host. Everything lives at https://domenhribernik.com, no www. The root .htaccess 301s www→apex and http→https (rules are host-anchored so local XAMPP is untouched). Every page also declares an absolute <link rel="canonical"> with a trailing slash, the same form the sitemap uses. One URL per page, everywhere.

robots.txt vs noindex. They answer different questions. robots.txt Disallow means "do not crawl": good for private tools, but a blocked page can still be indexed URL-only if someone links it. <meta name="robots" content="noindex"> means "you may crawl, do not index": the stronger promise, but only works if the crawler can fetch the page. Private tools here carry both. Never Disallow a page you want deindexed but that is already indexed: Google can no longer see the noindex. Remove the Disallow, let it recrawl, then block.

The sitemap. sitemap.xml at the root, referenced from robots.txt, regenerated by node tools/seo/generate.js (also run in CI on every deploy). Inventory comes from project-data.js plus tools/seo/config.js; a page gaining a noindex drops out automatically. lastmod comes from real git history: never fake freshness, Google learns to distrust a sitemap whose lastmod lies. Priorities: homepage 1.0, flagship 0.8, about 0.7, the rest 0.4.

Redirects. Permanent moves get a 301 (the old blog post.html?slug= URLs 301 to the static pages). Chains waste crawl budget; point old URLs at the final destination directly.

This site's machinery. After editing project-data.js, adding a blog post, or renaming a view: run node tools/seo/generate.js and commit what changed (sitemap, homepage fallback between the seo:projects markers, blog fallback, views/blog/<slug>/ pages). CI regenerates on deploy, so prod can never be stale, but committing keeps local and git truthful. Deleting a post leaves a stale copy on the server (deploys never delete); remove it over SFTP.

3. Core Web Vitals

Three field metrics, measured on real Chrome users (CrUX), checked with PageSpeed Insights:

  • LCP (Largest Contentful Paint) ≤ 2.5s: when the main content lands.
  • INP (Interaction to Next Paint) ≤ 200ms: how fast taps respond.
  • CLS (Cumulative Layout Shift) ≤ 0.1: how much the page jumps around.

Site-specific notes: the homepage intro overlay paints immediately (it is the LCP candidate itself, which is fine, it is content-sized and fast), but anything that delays it delays everything. The Tailwind CDN script is render-blocking JS on every view that uses it; that is the accepted cost of the no-build rule. Fonts load from Google Fonts with display=swap, so text never blocks.

What is worth fixing on a portfolio: images without dimensions (CLS), anything that blocks first paint, slow PHP endpoints on first render. What is not worth chasing: the last 200ms of LCP on a hobby view nobody queries. Check PSI quarterly for the flagship pages, record the result in the checklist's cwv column, and move on.

4. On-page

Titles. The pattern here is {Name} | {plain-language descriptor}, under ~60 characters so nothing truncates. The descriptor is what someone would actually type: "Plant Watering Schedule and Care Tracker" beats "Plant Care Manager". The homepage carries the money phrase: "Full-Stack Developer in Slovenia". Tarok's title is Slovenian because its searchers are.

Meta descriptions. Not a ranking factor; they are ad copy for the search result. 150 to 160 characters, front-load the value, write like the registry descriptions (lead with the problem, plain sentences). Google rewrites bad ones; give it no reason to.

Headings. Exactly one h1 per page that says what the page is, then a sane h2/h3 hierarchy. The checklist tracks the two current offenders (workout has none, spy has two).

Images. Meaningful alt text on content images; empty alt on decorative ones. Every image should have width/height or a CSS aspect ratio so it cannot shift layout.

Internal linking. Links are how authority flows through the site and how crawlers discover pages. The homepage fallback lists flagship projects first, so the strongest page passes its weight to the pages that should rank. When a blog post mentions a project, link the project. The about page carries the academic work. Nothing indexed should be more than two clicks from the homepage.

5. Structured data

JSON-LD tells machines what a page is, in schema.org vocabulary. It feeds rich results and gives AI engines clean facts to quote. What this site uses:

  • Person + WebSite on the homepage (one @graph), with sameAs links to GitHub, LinkedIn, Instagram. This is the identity anchor.
  • ProfilePage on about, embedding the same Person inline.
  • WebApplication on each flagship view (name, category, free Offer, author). Fits interactive tools better than SoftwareApplication.
  • Blog on the blog index; BlogPosting + BreadcrumbList on every generated post page (the generator emits these).
  • Organization on Everbloom, since it is a product, not a portfolio piece.

Each page's JSON-LD is self-contained: cross-page @id references do not resolve for Google, so the author is inlined as a small Person object everywhere.

Test with the Rich Results Test after changing any block. When NOT to add schema: types the page cannot honestly claim (Review, FAQ, HowTo without visible on-page counterparts). Dishonest markup earns manual actions, and Google ignores breadcrumbs with no visible breadcrumb UI, which is why only the generated posts carry BreadcrumbList.

6. Content and keyword strategy for a portfolio

Search intent first. Nobody searches "Domen Hribernik portfolio" except people who already know the name. Strangers arrive through queries the tools can win: a need ("tarok točkovanje", "print maze puzzles", "sourdough starter tracker") or a curiosity ("CSS 3D without WebGL"). Every public tool page should be written to answer one such query in its title and description.

The tarok case study. "Tarok točkovanje" is a real, low-competition Slovenian query with a tool-shaped answer, and this site has the tool. The page now targets it in Slovenian. The highest-leverage next step is one Slovenian blog post ("Kako šteti točke pri taroku") that explains the scoring rules in prose and links the tool: the post catches the informational query, the tool catches the transactional one, and they link to each other.

Blog topics with demand vs diary posts. Both are fine, they do different jobs. A diary post ("In Praise of Small Software") earns links and shows voice; a demand post answers something people type ("markdown blog without a build step"). Aim the title of a demand post at the query, not at cleverness. One post a month, of either kind, beats every other lever on this page.

Evergreen tool pages beat posts. A tool page ranks for years and improves with every feature; treat the flagship views as the site's permanent content and the blog as the thing that feeds them links and context.

7. Authority and backlinks for developers

Links are still the strongest ranking signal, and a personal site starts with none. The honest sources, in rough order of effort:

  • GitHub profile README: link the domain and two or three flagship projects. High-trust domain, takes ten minutes.
  • Profile links everywhere: LinkedIn website field, Instagram bio. Mostly nofollow, still identity-confirming for the knowledge graph.
  • dev.to republishing: repost blog entries with canonical_url set to the original (dev.to supports this natively), so the canonical credit flows home while the post reaches an audience.
  • Hacker News / Lobsters: submit essays as plain links. One front-page hour is worth more links than a year of directories.
  • Slovenian channels for tarok: r/Slovenia, slo-tech.com, tarok club pages. The audience is small and exactly right.

The quality bar: never directories-for-directories'-sake, never bought links, never reciprocal link pages. A handful of real links from real pages beats hundreds of junk ones, and junk can hurt.

8. AI search and GEO

Generative engines (ChatGPT search, Perplexity, Google AI Overviews) are now a meaningful referrer class. They select sources they can fetch, parse, and quote. What moves the needle:

  • Static HTML. Most AI crawlers do not execute JS. The prerendered fallbacks and static blog pages exist for exactly this reader.
  • Being quotable. Clear one-sentence definitions of what each thing is ("Nebo is a live map of the night sky computed in the browser"), stated in the page text, not only in a meta tag. Name your concepts; engines quote names.
  • Structured data and honest freshness (sections 2 and 5) double as GEO signals: they are the machine-readable statement of the same facts.
  • Bing matters again: ChatGPT browsing rides Bing's index, so Bing Webmaster Tools is no longer optional (import from GSC, one click).
  • llms.txt (a proposed root file listing what an LLM should read) is still a proposal without confirmed adoption by the major engines. Skip it until an engine documents that it reads one; revisit quarterly.

Checking your citations: ask the engines the queries you want to win ("tarok scoring app", "CSS-only 3D bouquet") and see whether the site is cited; GSC's performance report also breaks out AI Overview impressions. Record findings in the checklist notes.

9. Google Search Console workflow

One-time setup (do today):

  1. Add a Domain property for domenhribernik.com in Search Console (DNS TXT record at the registrar; covers http/https/www at once).
  2. Submit https://domenhribernik.com/sitemap.xml under Sitemaps.
  3. URL Inspection → Request Indexing for: the homepage, about, the blog index, both post URLs, and the seven flagship views. This is the only lever that speeds up a new site's first crawl.
  4. Bing Webmaster Tools: sign in, "Import from GSC", done. This also feeds ChatGPT.
  5. Verify the deploy landed: curl -I https://domenhribernik.com/robots.txt, the sitemap URL, and curl -I http://www.domenhribernik.com/ (expect 301).

Reading the reports. Pages (indexing) tells you what Google kept and why it rejected the rest; "Crawled, currently not indexed" on a new site mostly means "wait and earn a link". Performance shows queries vs pages: sort by impressions to find queries where the site appears but nobody clicks (title/description problem), and by position 5 to 15 to find pages one improvement away from page one.

Request indexing again after any significant page change; otherwise let the sitemap do its job.

10. Operating cadence

  • On every content change: run node tools/seo/generate.js, commit the artifacts. CI regenerates anyway; keep git truthful.
  • Monthly (15 minutes): GSC Pages + Performance skim; request indexing on anything new; flip the checklist's indexed cells that GSC confirms; one look at AI Overview impressions.
  • Monthly (the real work): one blog post. Nothing on this page compounds like content.
  • Quarterly: PSI on the flagships (record in cwv), a full-site audit with an audit skill (npx skills add seo-skills/seo-audit-skill or coreyhaines31/marketingskills@seo-audit, needs Node 20+), re-check the llms.txt adoption question, and reread this playbook against reality.

The checklist tab is the living state: cells cycle on click (in memory), and "Copy updated JSON" exports the new state to paste back into views/seo/checklist.json and commit. The test suite validates the file, so a bad paste fails node --test tests/ instead of silently breaking this page.
